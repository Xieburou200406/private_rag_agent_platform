一、统一全局返回码、鉴权规则、请求格式 完整释义（工业级后端标准，写进你的02_接口规范.md）
这套规范是Django DRF后端层对外所有REST接口强制统一标准，作用是：前后端、网关、Agent引擎交互时格式完全一致，不用每个接口单独协商，降低联调成本，是企业项目和学生Demo的核心分界线。
下面分三块讲清楚定义、落地写法、和你项目的关联。
一、统一请求格式（所有客户端→网关→Django的入参标准）
含义
系统内所有API请求，统一规定请求头、请求方式、传参位置、数据格式，任何接口都遵循同一套模板，不出现五花八门的传参方式。
1. 统一请求方式规范
查询类接口（查知识库、问答记录、任务列表）：GET，参数放URL Query
新增/提交类（上传文档、发起问答、创建智能体任务）：POST，参数放Body JSON
修改类（编辑知识库、修改权限）：PUT
删除类（删除文档、删除对话）：DELETE
2. 统一请求头固定字段（鉴权配套）
所有请求必须携带2个Header，缺少直接拦截返回错误码：
Authorization: Bearer {用户token}  # 鉴权凭证
Content-Type: application/json     # 强制JSON传输，禁止表单、文本
3. 统一Body请求体模板（POST/PUT接口强制）
所有提交数据统一外层包裹，扩展字段统一放extra，前端/Agent解析逻辑复用：
{
  "data": {},       // 核心业务参数（不同接口填不同内容）
  "extra": {}       // 扩展参数：分页、过滤、自定义配置
}
示例：上传文档发起解析任务请求体
{
  "data": {
    "doc_id": "doc_1001",
    "kb_id": "kb_005",
    "chunk_mode": "llm_semantic"
  },
  "extra": {
    "async": true,
    "batch_size": 10
  }
}
和你项目的关联
C++网关会校验Content-Type和Authorization头，非法格式直接拦截，不转发到Django；第三层Agent引擎调用后端接口时，也严格遵循这套请求格式，不用单独适配。
二、统一全局鉴权规则（身份校验标准）
含义
全系统一套身份校验逻辑，所有接口统一权限拦截逻辑，区分游客、普通用户、管理员，统一token签发、过期、权限校验流程，不会出现部分接口免登、部分接口校验逻辑不一样的混乱情况。
完整落地规则（可直接写入文档）
凭证标准
 用户登录Django后端后，Redis生成JWT Token，有效期2小时，存储用户ID、角色、知识库权限；所有请求Header携带Authorization: Bearer xxx鉴权。
网关前置拦截
 C++网关第一层先校验Token格式合法性，空Token/格式错误直接返回401，不转发业务请求，减轻后端压力。
后端二次权限校验
 Token合法后，Django根据Token内角色判断接口访问权限：普通用户：仅可查看自己的知识库、问答记录
管理员：全量知识库、用户管理、评测数据导出权限

免登白名单（仅少量接口）
 仅首页静态接口、登录接口放开鉴权；向量检索接口也走Token校验，防止非法调用向量库消耗算力。
Token过期/失效规则
 Token过期、用户注销、权限变更时，Redis主动拉黑Token，全局统一返回401错误。
和你项目的关联
防止非法人员调用Milvus向量库、批量消耗LLM额度，网关+后端双层鉴权，是企业私有化AI平台安全必备规范。
三、统一全局返回码 + 统一响应格式（所有接口输出固定模板）
含义
无论成功、失败、权限不足、参数错误、向量库超时、LLM调用失败，所有接口返回完全相同JSON结构，固定一套数字错误码，前端、C++网关、Agent引擎解析一套逻辑就能处理全部接口，不用单独适配。
1. 全局统一返回体模板（强制所有接口输出）
{
  "code": 200,       // 全局状态码，数字标识结果
  "msg": "操作成功", // 可读中文提示
  "data": {},        // 业务返回数据，无数据返回null
  "request_id": "req_xxxx" // 全局请求唯一ID，用于日志排查、性能统计
}
2. 全局统一返回码对照表（工业级通用，写入你的接口规范文档）
状态码	含义	触发场景（贴合你的项目）	
200	请求成功	问答成功、文档入库、查询数据正常	
400	参数非法	文档格式错误、向量检索参数缺失、Prompt参数错误	
401	未登录/Token失效	无Authorization头、Token过期、Token伪造	
403	权限不足	用户访问不属于自己的知识库、普通用户导出评测数据	
404	资源不存在	知识库ID不存在、对话记录已删除	
429	请求限流	C++网关QPS限流，高频调用向量库/LLM接口	
500	服务内部异常	LLM调用超时、Milvus向量库宕机、数据库崩溃	
502	下游服务不可用	Agent引擎离线、向量库服务断开	
504	请求超时	LLM生成回答超时、大批量文档解析超时	
3. 统一返回优势（面试加分点）
C++网关捕获502/429错误码，直接做熔断、重试逻辑；
第三层Agent引擎调用后端接口，只用一套代码解析返回结果，区分成功/权限/服务异常；
第四层评测模块批量请求接口，统一捕获错误码做失败样本统计；
前端页面全局统一弹窗提示，不用每个接口单独写报错逻辑。
四、三者结合对项目的整体价值（面试口述）
工程标准化，区分学生Demo
 绝大多数学生写RAG项目接口返回格式混乱、鉴权零散，你提前定义全局请求/响应/鉴权规范，是标准企业后端开发流程。
分层解耦，降低联调成本
 C++网关、Django后端、多智能体引擎三层互相调用，全部遵循同一套标准，跨语言交互无格式兼容问题。
安全可控
 统一鉴权+网关前置拦截，防止向量库、LLM接口被恶意刷取，贴合企业私有化AI平台安全需求。
便于日志、性能统计
 每个响应携带request_id，串联网关耗时、后端处理、LLM调用全链路日志，方便第四层评估模块做性能指标统计。
五、直接写入02_接口规范.md的固定段落模板
1. 全局请求规范
所有接口统一请求头：Content-Type: application/json、Authorization: Bearer {token}；
请求Method严格区分：GET查询、POST新增、PUT修改、DELETE删除；
POST/PUT接口统一JSON请求体，外层固定data业务参数、extra扩展参数。
2. 全局鉴权规范
全接口JWT Token鉴权，Token存储于Redis，携带用户ID、角色、知识库权限；
C++网关第一层校验Token格式，非法Token直接拦截返回401；
Django后端二次校验角色权限，区分管理员/普通用户接口访问范围；
Token过期、注销、权限变更实时拉黑，全局统一失效逻辑。
3. 全局统一响应规范
所有接口强制返回统一JSON结构：code、msg、data、request_id；
全局固定一套状态码，覆盖参数、权限、限流、LLM/向量库服务异常全场景；
网关、Agent引擎、前端根据code统一处理成功/失败逻辑，无需单独适配。




预定义四大模块接口清单（仅路径+功能，参数后续补充，直接写入docs/02_接口规范.md）
一、网关转发接口（C++网关对外暴露，统一流量入口）
所有请求统一前缀 /gateway/v1
POST /gateway/v1/vector/search：向量检索专用接口，直连Milvus/FAISS，跳过Django，实现低延迟检索
POST /gateway/v1/business/proxy：通用业务代理接口，转发全部Django后端业务请求
GET /gateway/v1/health：网关健康检查接口，用于Docker服务存活探测
GET /gateway/v1/stats/rt：获取网关实时性能耗时统计数据，供离线评测模块采集
二、Django业务接口（后端核心REST接口，前缀 /api/v1）
1. 用户权限模块
POST /api/v1/user/login：用户登录，下发JWT鉴权Token
POST /api/v1/user/logout：用户登出，Redis拉黑Token
GET /api/v1/user/info：查询当前登录用户信息、角色权限
GET /api/v1/user/kb/permission：查询用户可访问的知识库列表
2. 知识库管理模块
POST /api/v1/kb/upload：上传文档，发起文档解析入库任务
GET /api/v1/kb/list：分页查询用户所属知识库
GET /api/v1/kb/doc/list：查询指定知识库下全部文档元数据
DELETE /api/v1/kb/doc/remove：删除文档，同步清空向量库对应向量
3. 问答交互模块
POST /api/v1/chat/query：发起单轮问答请求，调用Agent引擎生成回答
GET /api/v1/chat/history：分页查询历史对话记录
POST /api/v1/chat/clear：清空当前用户对话上下文缓存
4. 智能体任务调度模块
POST /api/v1/agent/task/create：创建多智能体任务（数据分析/文档总结）
GET /api/v1/agent/task/list：分页查询用户全部智能体任务
GET /api/v1/agent/task/detail：查询单条任务执行结果、运行状态
5. 评测数据导出模块
GET /api/v1/eval/dataset/export：导出历史问答全量数据集，供给离线评估模块
GET /api/v1/eval/stats/query：导出网关、向量库响应耗时统计原始数据
三、Agent调度内部接口（Django远程调用，仅容器内网可访问，前缀 /agent/internal/v1）
POST /agent/internal/v1/doc/parse：文档解析Agent入口，完成文本清洗、语义分块、向量化入库
POST /agent/internal/v1/rag/chat：标准RAG问答流水线，执行Query扩展、向量召回、Rerank、答案生成
POST /agent/internal/v1/multi/agent/run：多智能体协同调度入口，基于Function Calling自动分配检索/分析/总结Agent
POST /agent/internal/v1/llm/label：轻量化LLM辅助接口，生成文档标签、任务标题、问题分类标签
GET /agent/internal/v1/health：Agent引擎服务健康探测接口
四、评测导出接口（第四层离线模块读取数据专用，Django提供，前缀 /api/v1/eval）
GET /api/v1/eval/dialog/dataset：批量拉取全量历史问答样本，构建标准化测试数据集
GET /api/v1/eval/vector/rt_log：批量拉取向量检索响应耗时、召回片段原始日志
POST /api/v1/eval/llm/batch_test：批量调用Agent引擎执行对照评测实验
GET /api/v1/eval/experiment/report：读取已完成实验的指标统计原始数据，用于绘图可视化

## 六、容器内网服务端口与路由对照（与 deploy 配置对齐）

| 服务 | 容器名 | 内网端口（.env） | 对外入口 |
|------|--------|------------------|----------|
| Nginx | nginx | 80 | `http://localhost:${NGINX_HTTP_PORT}` |
| C++ 网关 | cpp_gateway | `GATEWAY_LISTEN_PORT=8080` | `/gateway/v1/*` 经 Nginx 转发 |
| Django | django_backend | `DJANGO_LISTEN_PORT=8000` | 经网关 `/gateway/v1/business/proxy` 代理 |
| Agent 引擎 | agent_engine | `AGENT_LISTEN_PORT=8001` | 仅内网，前缀 `/agent/internal/v1` |
| MySQL | mysql | 3306 | 仅内网 |
| Redis | redis | 6379 | 仅内网，密码 `REDIS_PASSWORD` |
| Milvus | milvus | 19530 | 仅内网 |

**注意：** Nginx 配置中 upstream 端口须与 `.env` 默认值一致；若修改 `GATEWAY_LISTEN_PORT` / `DJANGO_LISTEN_PORT`，需同步修改 `deploy/nginx/nginx.conf`。
