# 04 部署运维手册（私有 RAG 四层架构平台）

## 一、部署产出总览

| 产出物 | 路径 | 状态 |
|--------|------|------|
| 容器编排 | `deploy/docker-compose.yml` | C++ 网关 / Django / Agent / MySQL / Redis / Milvus / Nginx |
| 环境变量 | `deploy/.env.example` → `.env` | DB、Redis、Milvus、LLM、限流参数全部外置 |
| 反向代理 | `deploy/nginx/nginx.conf` | `/gateway/v1/` → 网关，`/static/` → Django |
| Redis 配置 | `deploy/redis/redis.conf` | 持久化 + 内存淘汰策略 |
| 数据库初始化 | `deploy/init_sql/init.sql` | 首次启动自动建表 |
| 运维脚本 | `deploy/scripts/*.sh` | 文档导入 / 日志清理 / 离线打包 / 一键重启 |
| 分层 Dockerfile | 各模块目录 | 单模块写完即可独立容器调试 |

> 向量库采用 **Milvus standalone**（架构文档已用 Milvus 替代 FAISS 容器）。

## 二、前置环境校验（启动前必做）

1. Docker Desktop 正常运行（Windows 建议开启 WSL2 后端）
2. 镜像拉取：配置国内镜像加速器或代理，避免 `docker.io` 超时
3. 复制环境变量模板并填写密钥：

```powershell
cd deploy
copy .env.example .env
# 编辑 .env：MYSQL_ROOT_PASSWORD、REDIS_PASSWORD、LLM_API_KEY、DJANGO_SECRET_KEY
```

4. **Windows 注意**：`.sh` 脚本请在 **Git Bash** 或 **WSL** 中运行，PowerShell 不支持 `chmod`/`bash`
5. `deploy/redis/redis.conf` 须为 **LF 换行**（已配置 `.gitattributes`，Windows 误改 CRLF 会导致 Redis 启动失败）
6. **C++ 网关依赖 Linux epoll**，Windows 原生 PowerShell 无法直接 `cmake && make`，请用 Docker 或 WSL2 编译（见第五节）

## 三、一键部署完整流程

### 3.1 构建并启动

```powershell
cd deploy
docker compose build
docker compose up -d
docker compose ps
```

期望：`nginx`、`mysql`、`redis`、`milvus`、`cpp_gateway`、`django_backend`、`agent_engine` 均为 **Up**。

### 3.2 验证容器内网互通

**方式 A：在 Django 容器内测试**

```powershell
docker exec -it django_backend bash
ping -c 2 mysql
ping -c 2 redis
ping -c 2 milvus
curl -s http://cpp_gateway:8080/gateway/v1/health
curl -s http://agent_engine:8001/agent/internal/v1/health
curl -s http://127.0.0.1:8000/api/v1/health
exit
```

**方式 B：宿主机经 Nginx 测试**

```powershell
# PowerShell 中 curl 是 Invoke-WebRequest 别名，建议用 curl.exe 避免交互警告
curl.exe -s http://localhost/health
curl.exe -s http://localhost/gateway/v1/health
```

全部返回 JSON / `ok` = 基础网络链路打通。

### 3.3 平台访问入口

| 入口 | 地址 |
|------|------|
| Nginx 统一入口 | `http://localhost:80` |
| 网关 API | `http://localhost:80/gateway/v1/*` |
| Django 业务（经网关代理） | `http://localhost:80/gateway/v1/business/proxy/api/v1/*` |
| Django 内网直连（调试） | `http://django_backend:8000/api/v1/health` |

### 3.4 离线评测模块（默认不启动）

```powershell
cd deploy
docker compose --profile eval run --rm evaluation_module
```

## 四、自动化运维脚本

脚本目录：`deploy/scripts/`，请在 **Git Bash / WSL** 中从 `deploy/` 目录执行。

### 1. 批量文档导入

```bash
cd deploy
bash scripts/batch_import_docs.sh
```

调用 Django 容器内 `scripts/batch_import_kb.py`（当前为占位，待接入 Agent 引擎）。

### 2. 日志清理（7 天前）

```bash
cd deploy
bash scripts/clean_log.sh
```

### 3. 离线打包

```bash
bash deploy/scripts/package_project.sh
```

导出基础镜像 `deploy/all_base_images.tar` 及源码压缩包，无网环境先 `docker load -i all_base_images.tar` 再 `docker compose up -d`。

### 4. 一键重启

```bash
cd deploy
bash scripts/restart_all.sh
```

## 五、分模块独立调试

### C++ 网关（Windows / Linux）

**推荐：Docker 内编译运行（Windows 适用）**

```powershell
cd deploy
docker compose build cpp_gateway --no-cache
docker compose up -d cpp_gateway
docker compose logs -f cpp_gateway
```

**Linux / WSL2 本地编译**

```bash
cd cpp_gateway
mkdir -p build
cd build
cmake ..
make -j4
# 确保 deploy/.env 存在
GATEWAY_ENV_FILE=../deploy/.env ./gateway_server
```

**不要在 PowerShell 5.x 中使用 `&&` 链式命令**（会报 `InvalidEndOfLine`），应改用：

```powershell
cd cpp_gateway
New-Item -ItemType Directory -Force -Path build | Out-Null
Set-Location build
cmake ..
cmake --build . -j 4
```

> 即使编译成功，`gateway_server` 也仅能在 Linux/WSL 运行（epoll 为 Linux 专有 API）。

开发阶段容器内运行 `/usr/local/bin/gateway_server`（epoll 框架）；源码挂载在 `/app`，改代码后需 `docker compose build cpp_gateway` 重新编译镜像。

### Django / Agent

```powershell
cd deploy
docker compose build django_backend agent_engine
docker compose up -d django_backend agent_engine
```

## 六、故障排查

### 6.1 容器与中间件

| 现象 | 原因 | 解决 |
|------|------|------|
| `invalid spec: :/app/logs` | 旧版 compose 用了空 `VOLUME_*` 变量 | 使用当前 compose 内置命名卷 |
| Redis `Bad directive rdbfilename` | Redis 7 改为 `dbfilename`；或 CRLF 换行 | 使用 LF 换行 + `dbfilename dump.rdb` |
| `ModuleNotFoundError: rag_project` | Django 骨架缺失 | 已补全 `backend_django/rag_project/` |
| cpp_gateway 反复 Restarting | 旧 CMD `/bin/bash` 立即退出 | 已改为 `gateway_server` 常驻进程 |
| cpp_gateway 找不到 .env | 容器内未挂载 deploy/.env | compose 已挂载 `./.env:/app/deploy/.env:ro` |
| `chmod` 在 PowerShell 不可用 | Windows 无此命令 | 用 Git Bash / WSL 运行 `.sh` |
| gcc:13 `sources.list` 不存在 | Debian 新版源文件路径变化 | Dockerfile 已兼容 `debian.sources` |
| django_backend 连不上 MySQL | 密码与 `.env` 不一致 | 检查 `MYSQL_ROOT_PASSWORD` |

### 6.2 Windows PowerShell 常见误用

| 现象 | 原因 | 解决 |
|------|------|------|
| `` `&&` 不是有效语句分隔符 `` | PowerShell 5.x 不支持 `cmd` 风格 `&&` | 分行执行，或用 `;` 分隔；推荐 Git Bash / WSL |
| `mkdir -p build && cd build` 报错 | 同上 | `New-Item -Force build; Set-Location build` |
| `curl` 弹出 Script Execution Risk | PowerShell 中 `curl` 是 `Invoke-WebRequest` 别名 | 使用 `curl.exe -s http://localhost/health` |
| `git commit -m "$(cat <<'EOF'..."` 失败 | PowerShell 不支持 bash HEREDOC | 直接用：`git commit -m "deploy: 完善 docker-compose 编排"` |
| 粘贴整段教程到终端乱码/报错 | 把 Markdown 说明文字当作命令执行 | 只复制代码块内的命令，不要粘贴说明段落 |
| 本地 `cmake` 通过但无法运行 | epoll 仅 Linux 支持 | Windows 请用 `docker compose build cpp_gateway` |

## 七、生产环境建议

- C++ 网关：多阶段构建，剥离 gcc 编译环境，CMD 改为 `./build/gateway_server`
- 密钥：`.env` 已在 `.gitignore`，禁止提交真实密钥
- 定时任务：Linux crontab / Windows 计划任务每日执行 `clean_log.sh`
- 评测模块：业务低峰期手动 `--profile eval run`，不占实时算力

## 八、架构配套说明

- 四层目录：`cpp_gateway` / `backend_django` / `agent_engine` / `evaluation_module`
- 向量检索：C++ 网关直连 Milvus，跳过 Django（`VECTOR_SKIP_PROXY=true`）
- 鉴权链路：Nginx → C++ 网关 Token 拦截 → Django 二次权限校验
- 每写完一个模块，挂载卷热更新代码，`docker compose restart <service>` 即可调试
